# Maestro - Base de Conhecimento Interna (BCI)

> **Fonte da Verdade para a Arquitetura Cognitiva do Sistema Maestro.**

## 1.0 Propósito Deste Documento
Este inventory_bci.md é a documentação fundamental e a fonte da verdade para o repositório da Base de Conhecimento Interna (BCI) do Maestro. Ele é projetado para fornecer um guia de inventário de todos os componentes do Maestro.

## 2.0 Filosofia Arquitetural: "Knowledge as Code"
A Base de Conhecimento Interna (BCI) do Maestro opera sob a filosofia de "Knowledge as Code". Isso significa que o conhecimento jurídico, estratégico e processual não é apenas armazenado, mas estruturado, versionado e gerenciado com o mesmo rigor de um código-fonte de software. Cada arquivo `.md` é um módulo de conhecimento autocontido, com metadados explícitos que definem sua identidade, versão e, crucialmente, suas interdependências.

Essa abordagem garante que a BCI seja:
- **Modular:** O conhecimento é encapsulado em componentes discretos e reutilizáveis.
- **Versionada:** Alterações no conhecimento são rastreáveis e gerenciáveis.
- **Interconectada:** O sistema de dependências permite a um orquestrador externo reconstruir o contexto completo necessário para qualquer tarefa, garantindo que a análise seja sempre holística e informada por todos os componentes relevantes.
- **Auditável:** A estrutura clara e os metadados explícitos permitem uma verificação sistemática da integridade e completude da base.

## 3.0 Anatomia de um Módulo de Conhecimento
Cada arquivo `.md` na BCI é um "Módulo de Conhecimento" e segue uma estrutura padrão:

1.  **Frontmatter (YAML):** Um bloco de metadados no início do arquivo, delimitado por `---`. Ele contém campos essenciais:
    *   `id`: Um identificador único e legível para o módulo.
    *   `version`: A versão semântica do módulo.
    *   `type`: O tipo de componente de conhecimento (e.g., `core_concept`, `module`, `pillar`).
    *   `dependencies`: Uma lista de outros `id`s de módulos dos quais este módulo depende. Este é o campo mais crítico para a reconstrução do contexto.

2.  **Corpo (Markdown):** O conteúdo principal do módulo, escrito em Markdown. Ele contém a "Descrição da Finalidade" e a especificação detalhada do conhecimento que o módulo encapsula.

## 5.0 Guia de Contribuição
A manutenção e expansão desta Base de Conhecimento devem seguir rigorosamente os princípios de "Knowledge as Code".

1.  **Estrutura do Módulo:** Todo novo módulo `.md` DEVE conter um `frontmatter` YAML válido com `id`, `version`, `type`, e `dependencies`.
2.  **Atomicidade:** Cada módulo deve focar em um único conceito, princípio ou framework de forma coesa.
3.  **Dependências Explícitas:** Todas as dependências de um módulo em outro DEVEM ser declaradas no campo `dependencies`.
4.  **Validação:** Após qualquer modificação, o processo de auditoria e verificação (conforme executado para gerar este `README.md`) deve ser re-executado para garantir a integridade contínua do sistema.
